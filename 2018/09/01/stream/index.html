redis unexpected end of stream 问题

发表于 2018-09-01 | 分类于 开发工具 | | 阅读次数:

使用redisTemplate来进行java项目的开发的时候log一直遇见一个问题，搜了下
基本上都是要我去修改下使用的redis的config文件，设置关闭缓冲区限制，

client-output-buffer-limit normal 0 0 0

client-output-buffer-limit slave 0 0 0

client-output-buffer-limit pubsub 0 0 0

但是这样还是没能解决我的问题，又看到许多人说的设置time’out 设置长一点或者干脆设置不过期，我运行之后还是有这个unexpected end of stream的问题。但是我用RedisDesktopManagement查询key是可以查出来结果的，
我只能从代码端入手看怎么产生这个问题的

解决

仔细看了下spring-redis的xml和redis的配置文件，然后重新走了遍流程，发现redisTemperate的exist方法返回有问题，当我删除key之后再调用exist的方法直接报了异常，导致我if判断返回的是一个异常，它还头铁的把我返回为true的方法，也就是delkey方法执行了，（这个地方应该是没有做异常捕捉导致的），我然后修改了下redisUtil的封装方法，将exist的方法异常时返回false，然后在调用的这里catch异常。但是令我感到意外的是，redisTemplate的事务在这里没有起到作用，于是我去查了下,
通过

private boolean enableTransactionSupport = false;

可以看出RedisTemplate是默认事物关闭的。 在RedisTemplate源码中搜索该变量，发现

也就是如果设置参数enableTransactionSupport=true,则系统自动帮我们拿到了事务中绑定的连接。可以在一个方法的多次对Redis增删该查中，始终使用同一个连接。这里我猜测是上一个delkey方法的事务的连接时间太长，导致新的请求发送redis还是使用的未delkey的redis，数据出现了误差，所以导致unexpected end of stream 的问题

  • 解决方式:

    删除spring-redis配置中的enableTransactional 或者 在transactional方法中手动关闭连接，令其返回连接池。
